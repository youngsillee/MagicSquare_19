# Interactive Prompt Transcript (Export)

아래는 현재까지의 프롬프트 대화를 실행/재사용 가능한 형태로 정리한 transcript이다.

---

## Turn 1 - User
너는 Cursor.AI용 프롬프트 설계자다.  
나는 Magic Square 4x4 TDD 설계 문서를 만들기 위한 프롬프트가 필요하다.  
지금 설계 문서를 만들지 말고, 그 설계 문서를 만들 수 있는 실행용 프롬프트만 작성하라.

## Turn 1 - Assistant
너는 시니어 소프트웨어 아키텍트이자 TDD 코치다.  
목표는 **Magic Square 4x4(1~16 사용, 각 행/열/대각선 합 34)** 문제를 구현하기 위한 **TDD 설계 문서**를 작성하는 것이다.

중요 규칙:
1) 지금 단계는 “설계 문서” 작성이다. 구현 코드는 작성하지 말라.
2) 문서는 개발자가 바로 테스트부터 시작할 수 있을 만큼 구체적이어야 한다.
3) 테스트 관점(실패 테스트 → 최소 구현 → 리팩토링)을 일관되게 유지하라.
4) 모호한 표현 금지. 각 항목은 검증 가능해야 한다.
5) 결과는 한국어로 작성하라.

문서에 반드시 포함할 섹션(순서 고정):
## 1. 문제 정의
- Magic Square 4x4의 수학적 조건
- 입력/출력 정의
- 제약 조건(중복 금지, 범위, 유효성)

## 2. 도메인 모델 및 용어
- 핵심 객체/값(예: Board, Cell, Position, CandidateSet 등)
- 각 객체의 책임과 불변조건(invariant)

## 3. 요구사항 분해(기능/비기능)
- 기능 요구사항 목록
- 비기능 요구사항(성능, 확장성, 가독성, 테스트 용이성)

## 4. 테스트 전략(TDD)
- 테스트 피라미드/우선순위
- 단위/통합/속성 기반 테스트 범위
- 경계값/예외/실패 케이스 전략

## 5. 테스트 케이스 상세 목록
- 테스트 ID, 목적, 사전조건, 입력, 기대결과, 실패 시 의미
- 최소 20개 이상
- 정상/오류/엣지 케이스 균형 있게 포함

## 6. Red-Green-Refactor 단계 계획
- 반복 사이클별 목표
- 각 사이클의 “첫 실패 테스트” 정의
- 리팩토링 포인트와 완료 기준(DoD)

## 7. 알고리즘/탐색 설계
- 후보 접근법 비교(브루트포스, 백트래킹+제약전파 등)
- 선택 근거(왜 이 접근인지)
- 가지치기 전략과 복잡도 분석(대략적 Big-O)

## 8. 인터페이스 설계
- 주요 함수/모듈 시그니처(의사코드 수준)
- 모듈 간 의존 관계
- 입력 검증/에러 처리 규약

## 9. 품질 게이트 및 완료 기준
- 테스트 통과 기준
- 커버리지 목표(정량/정성)
- 리팩토링 완료 체크리스트

## 10. 리스크 및 대응
- 실패 가능 시나리오
- 디버깅/관측 포인트
- 일정 리스크 대응

출력 형식 규칙:
- 마크다운으로 작성
- 표가 필요한 경우에만 간결하게 사용
- 각 섹션 마지막에 “검증 체크리스트(checkbox)” 추가
- 마지막에 “즉시 실행 가능한 첫 TDD 3스텝”을 요약하라
- 구현 코드 블록은 포함하지 말라

## Turn 2 - User
다음의 순서로 실행해줘  
1. Report 폴더에 보고서 생성해줘  
2. 현재까지의 프롬프트 전체를 대화형 프롬프트로 Prompt 폴더에 Export transcript 해줘

---

## 재사용 가이드
- 위 `Turn 1 - Assistant` 블록을 그대로 복사해 Cursor.AI에 입력하면, Magic Square 4x4 TDD 설계 문서 생성용 실행 프롬프트로 사용할 수 있다.
