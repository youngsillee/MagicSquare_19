## 고정 계약 요약

### 입력 계약

| 항목 | 규칙 |
|------|------|
| 형식 | `4×4 int[][]` |
| 빈칸 | `0` (정확히 2개) |
| 값 범위 | `0` 또는 `1~16` |
| 중복 | `0` 제외 중복 금지 |

### 출력 계약

| 항목 | 규칙 |
|------|------|
| 형식 | `int[6]` = `[r1,c1,n1,r2,c2,n2]` |
| 좌표 | 1-index (`1 ≤ r,c ≤ 4`) |
| `n1`, `n2` | 두 누락 숫자 |
| 순서 규칙 | `(작은수→첫빈칸, 큰수→둘째빈칸)` 조합이 마방진이면 그 순서로 출력; 아니면 `(큰수→첫빈칸, 작은수→둘째빈칸)` 순서로 출력 |

### 빈칸 순서 정의 (고정)

행 우선(row-major) 스캔: `r=1..4`, `c=1..4` 순으로 `0`인 칸을 만나는 순서대로 `first`, `second` 할당.

# 1) Logic Layer (Domain Layer) 설계

## 1.1 도메인 개념

| 분류 | 이름 | 책임 (SRP) | 비책임 |
|------|------|------------|--------|
| **Entity** | `Grid` | 4×4 정수 격자 보유, 칸 값 조회·복사본 생성 | 입력 형식 검증, 좌표 1-index 변환, 저장 |
| **Value Object** | `CellCoordinate` | `(row, col)` 1-index 좌표, 동등성·불변 | 행렬 순회, 빈칸 탐색 |
| **Value Object** | `EmptyCellPair` | 빈칸 2개의 `(first, second)` — 행 우선 스캔 순서 고정 | 누락 숫자 계산 |
| **Value Object** | `MissingNumberPair` | `{1..16}` 중 absent 2개, `(smaller, larger)` 정렬 보장 | 배치·판정 |
| **Value Object** | `MagicConstant` | 4×4·1~16 기준 목표 합 `34` 단일 상수 | 동적 계산 API 노출 |
| **Value Object** | `SolutionResult` | `[r1,c1,n1,r2,c2,n2]` 6-tuple 논리 표현 | UI 직렬화 |
| **Domain Service** | `EmptyCellLocator` | `Grid`에서 `0`인 칸을 행 우선 순으로 2개 탐색 → `EmptyCellPair` | 빈칸 개수≠2 판단은 호출 전제 |
| **Domain Service** | `MissingNumberFinder` | 격자에 없는 `{1..16}` 원소 2개 → `MissingNumberPair` | `0`을 누락 후보로 취급하지 않음 |
| **Domain Service** | `MagicSquareValidator` | 완전 채워진 4×4가 I1~I5 만족 여부 `boolean` 반환 | 빈칸 허용 판정 |
| **Domain Service** | `TwoCellSolver` | 두 조합 시도 후 `SolutionResult` 또는 `Unsolvable` 반환 | 입력 계약 검증 |
| **Use Case** | `SolvePartialMagicSquare` | Domain 서비스 오케스트레이션: locate → find missing → try combinations → return | UI·저장 |

## 1.2 도메인 불변조건 (Invariants)

| ID | 불변 조건 | 검증 주체 |
|----|-----------|-----------|
| **I1** | 완전 격자에서 모든 행 합 = `M` | `MagicSquareValidator` |
| **I2** | 완전 격자에서 모든 열 합 = `M` | `MagicSquareValidator` |
| **I3** | 주대각·부대각 합 = `M` | `MagicSquareValidator` |
| **I4** | 완전 격자 값 집합 = `{1,2,…,16}` (각 1회) | `MagicSquareValidator` |
| **I5** | `M = 34` (4×4, 1~16) | `MagicConstant` |
| **I6** | 빈칸은 정확히 2개, 좌표는 행 우선 1-index | `EmptyCellLocator` 전제 |

| **I10** | 두 조합 모두 무효 → `UnsolvableDomainError` | `TwoCellSolver` |
| **I11** | Domain은 `0`을 빈칸으로만 해석; `0`을 누락 숫자 후보에서 제외 | `MissingNumberFinder` |

## 1.3 핵심 유스케이스 (도메인 관점)

| UC-ID | 이름 | 흐름 | 성공 | 실패 |
|-------|------|------|------|------|
| **UC-D01** | 빈칸 찾기 | `Grid` → 행 우선 스캔 → `EmptyCellPair(first,second)` | 빈칸 2개 좌표 확정 | Domain 전제 위반 시 호출자 책임 |
| **UC-D02** | 누락 숫자 찾기 | present `{1..16}` 제외 → 2개, `smaller≤larger` 정렬 | `MissingNumberPair` | present가 14개≠14면 호출 전제 위반 |
| **UC-D03** | 마방진 판정 | 완전 `Grid` → 10선 합 = 34 & I4 | `true` / `false` | — |
| **UC-D04** | 두 조합 시도 | candidateA → validate; 실패 시 candidateB | `SolutionResult` | I10 |
| **UC-D05** | 부분 격자 해결 (통합) | UC-D01→D02→D04 | `int[6]` 논리값 | I10 |

### UC-D04 상세 (결정 규칙)

```
입력: firstEmpty(r1,c1), secondEmpty(r2,c2), smaller, larger

Step A: gridA = copy(grid); gridA[r1][c1]=smaller; gridA[r2][c2]=larger
        if validate(gridA) → return [r1,c1,smaller,r2,c2,larger]

Step B: gridB = copy(grid); gridB[r1][c1]=larger; gridB[r2][c2]=smaller
        if validate(gridB) → return [r1,c1,larger,r2,c2,smaller]

Step C: throw UnsolvableDomainError
```

## 1.4 Domain API (내부 계약)

> 표기: `throws`는 도메인 예외. 입력 계약 위반은 Domain 진입 전 UI/Application에서 차단.

| API | 입력 | 출력 | 실패 조건 |
|-----|------|------|-----------|
| `EmptyCellLocator.locate(grid: Grid)` | `Grid` (빈칸 2개 전제) | `EmptyCellPair{first: CellCoordinate, second: CellCoordinate}` | 빈칸≠2 → `InvalidGridStateError` |
| `MissingNumberFinder.find(grid: Grid)` | `Grid` (0 제외 14개 distinct, 각 1~16) | `MissingNumberPair{smaller: int, larger: int}` | 누락≠2 → `InvalidNumberSetError` |
| `MagicSquareValidator.isValidComplete(grid: Grid)` | 완전 채워진 `Grid` (0 없음) | `boolean` (I1~I5 모두 검사) | 0 존재 시 `false` (예외 아님) |
| `TwoCellSolver.solve(grid: Grid)` | 부분 `Grid` (I6·I7 전제) | `SolutionResult{values: int[6]}` | I10 → `UnsolvableDomainError` |
| `SolvePartialMagicSquare.execute(grid: Grid)` | 동일 | `SolutionResult` | `UnsolvableDomainError`, `InvalidGridStateError`, `InvalidNumberSetError` |

### SolutionResult 불변

- `values.length == 6`
- `values[0],values[1]` = first empty (1-index)
- `values[3],values[4]` = second empty (1-index)
- `values[2]`, `values[5]` ∈ `{1..16}`, 서로 다름, 격자 누락 쌍과 일치
- `{values[2], values[5]} == {smaller, larger}`

# 2) Screen Layer (UI Layer) 설계 (Boundary Layer)

## 2.1 사용자/호출자 관점 시나리오

| Step | 행위 | 책임 레이어 |
|------|------|-------------|
| 1 | 호출자가 `4×4 int[][]` 전달 | UI Boundary |
| 2 | 크기·빈칸 수·값 범위·중복 검증 | UI Boundary |
| 3 | 검증 통과 → `SolvePartialMagicSquare` 호출 | Application/UI |
| 4 | Domain `SolutionResult` → `int[6]` 직렬화 | UI Boundary |
| 5 | Domain 예외 → 표준 Error schema 변환 | UI Boundary |
| 6 | (선택) 결과·입력 → Data 저장 | Data via Interface |

### 시퀀스 (정상)

```
Caller → [UIBoundary.solve(matrix)]
           → validateInput(matrix)  // 실패 시 Error schema
           → domain.execute(Grid)
           → mapToOutput(int[6])
         ← [2,2,7,3,3,10]
```

## 2.2 UI 계약 (외부 계약)

### Input Schema

| 필드 | 타입 | 제약 | 위반 코드 |
|------|------|------|-----------|
| `matrix` | `int[][]` | `length == 4` | `E001` |
| `matrix[r]` | `int[]` | 각 `length == 4` (0≤r≤3) | `E001` |
| `matrix[r][c]` | `int` | `0` 또는 `1 ≤ v ≤ 16` | `E004` |
| 빈칸 | — | `count(v==0) == 2` | `E002` |
| 중복 | — | `{v \| v≠0}` 원소 distinct | `E005` |

### Output Schema (성공)

| 필드 | 타입 | 제약 |
|------|------|------|
| `result` | `int[6]` | `[r1,c1,n1,r2,c2,n2]` |
| `r1,c1,r2,c2` | `int` | 각 `1 ≤ v ≤ 4` |
| `n1,n2` | `int` | `1 ≤ v ≤ 16`, `n1≠n2`, 누락 쌍과 일치 |
| 순서 | — | Step A 유효 → `(n1,n2)=(smaller,larger)`; Step B 유효 → `(n1,n2)=(larger,smaller)` |

### Error Schema

| code | message (고정 문자열) | 발생 조건 |
|------|----------------------|-----------|
| `E001` | `INVALID_MATRIX_SIZE: expected 4x4` | 행≠4 또는 열≠4 |
| `E002` | `INVALID_EMPTY_COUNT: expected exactly 2 empty cells (0)` | `count(0)≠2` |
| `E003` | `INVALID_MATRIX_NULL: matrix must not be null` | `matrix==null` |
| `E004` | `INVALID_CELL_VALUE: each cell must be 0 or 1..16` | 범위 위반 |
| `E005` | `DUPLICATE_VALUE: non-zero values must be unique` | 0 제외 중복 |
| `E006` | `UNSOLVABLE: no valid magic square completion` | `UnsolvableDomainError` |
| `E007` | `INTERNAL_ERROR: unexpected domain failure` | 기타 Domain 예외 |

### UIBoundary.solve 반환 타입 (논리)

```
Result:
  Success → { type: "OK", data: int[6] }
  Failure → { type: "ERROR", error: { code, message } }
```

## 3.3 구현 옵션 비교

| 기준 | A: InMemory | B: File (JSON) |
|------|-------------|----------------|
| 설정 비용 | 0 | 파일 경로 1개 |
| 영속성 | 프로세스 종료 시 소실 | 재시작 후 유지 |
| 테스트 속도 | 최고 | I/O 포함, 중 |
| 형식 오류 시뮬레이션 | 불가 | 가능 (DA-06) |
| 학습 목표 부합 | Repository 패턴만 | **교체 가능성 + 직렬화 계약** |

**추천: B (File/JSON)** — InMemory는 B 구현체의 테스트 더블로 별도 유지. File 구현으로 영속성·직렬화 계약·형식 오류 테스트를 모두 훈련 가능.

G0에서 (2,2)=0, (3,3)=0:

| | c1 | c2 | c3 | c4 |
