## STEP 1 — Observation (관찰)

### 배경

워크스페이스(`MagicSquare_1004`)에는 코드·문서가 없는 상태에서, 프로젝트 이름과 요청을 바탕으로 문제를 관찰했다.

### 1. 현재 우리가 해결하려는 상황은 무엇인가?

**16개의 서로 다른 정수(보통 1~16)를 4×4 격자에 배치했을 때, 행·열·대각선마다 합이 같아지는 배치를 찾거나 검증·생성하는 상황**이다.

관찰 관점:

| 요소 | 내용 |
|------|------|
| **입력** | 16개의 숫자(또는 "1부터 16까지 사용" 제약) |
| **구조** | 4행 4열의 이차원 배열 |
| **제약** | 각 칸에 숫자는 한 번씩만; 4개 행 + 4개 열 + 2개 대각선 = **10개 선**의 합이 모두 동일 |
| **미지** | "어떤 배치가 이 제약을 만족하는가?" 또는 "주어진 배치가 조건을 만족하는가?" |

단순히 "표를 채운다"가 아니라, **여러 선(line)에 걸친 합 일치라는 구조적 제약을 동시에 만족하는 배치**를 다루는 상황이다.

### 2. 왜 4×4 마방진 문제를 다루는가?

| 관찰 | 의미 |
|------|------|
| **규모가 작다** | 3×3은 유일해(회전·대칭 제외) 거의 하나, 5×5 이상은 복잡도 급증 |
| **패턴이 존재한다** | 4×4는 학습 가능한 구성 규칙이 있어 탐색 없이도 답을 만들 수 있음 |
| **검증이 명확하다** | 10개 선의 합만 비교하면 정답 여부를 기계적으로 판별하기 쉬움 |
| **확장 여지** | 생성 / 검증 / 사용자 입력 / 시각화 등 단계적 요구 축적에 적합 |

프로젝트명 `MagicSquare_1004`는 **4×4에 초점을 둔 학습·실습 프로젝트**로 해석 가능하다.

### 3. 학습·시스템 설계 맥락

**학습 맥락**

- 2차원 배열, 인덱스, 순회, 불변 조건 검증
- 생성(constructive) / 검증(validation) / 탐색(search) 패턴
- 마법常数(magic constant): 4×4, 1~16이면 `(1+…+16)/4 = 34`
- 입력 → 처리 → 출력의 관심사 분리

**시스템 설계 맥락**

- 도메인 모델: Grid, Cell, Line(행/열/대각), MagicConstant
- 유스케이스: 자동 생성, 사용자 격자 검증, (확장) 힌트·오류 표시
- 품질 속성: 정확성, 결정성, 확장성

### 관찰 관점 종합

> **"1부터 16까지의 서로 다른 정수를 4×4 격자에 놓되, 네 행·네 열·두 대각선 — 열 개의 선 — 각각의 합이 모두 34가 되도록 하는 배치를 찾고, 그 배치가 조건을 만족하는지 확인할 수 있는 프로그램을 만들려는 상황"**

**관찰의 핵심**

1. **객체**: 16칸 격자 + 10개의 "합을 맞춰야 하는 선"
2. **제약**: 숫자 중복 없음 + 모든 선의 합 동일
3. **불확실성**: 생성만 / 검증만 / 둘 다 — 아직 요구 미명시
4. **맥락**: 교육·실습용, 4×4 고정, 구조적 제약 만족 문제의 프로그램화

## STEP 2 — Why #1

### 질문

**왜 마방진을 완성해야 하는가?**

### 가정 (A)

> **"4×4 격자에 1~16을 배치할 때, 행·열·대각선 합이 모두 같아지는 '완성된 배치'를 얻어야, 그 배치가 실제로 존재하고 조건을 만족한다는 것을 확인·학습·공유할 수 있기 때문이다."**

완성의 목적은 퍼즐의 "승리"가 아니라, **구조적 제약(10개 선의 합 일치)을 동시에 만족하는 유효한 상태를 하나 확보하는 것**이다.

### 가정 A에서 드러나는 불편함

| # | 불편함 | 설명 |
|---|--------|------|
| 1 | **인지 부하** | 10개 선의 합을 동시 추적; 한 칸 변경 시 3~4개 선 동시 영향 |
| 2 | **검증 비용** | "16칸 채움" ≠ "조건 만족"; 10번 합산·비교 필요 |
| 3 | **탐색 공간** | 16!에 가까운 배치 중 유효한 것은 극소; 규칙 없으면 재현 어려움 |
| 4 | **다중 해** | 회전·대칭 포함 여러 유효 배치; "하나의 정답" 착각 위험 |
| 5 | **오류 전파** | 초기 오배치 → 데드락; 되돌림·상태 추적 없으면 처음부터 재시작 |
| 6 | **목적 혼선** | 학습 / 데모 / 퍼즐 목적 혼재 → 생성·검증·힌트·정답 노출 요구 충돌 |

### 구조적 문제 요약

```
[목표: 10개 선의 합 일치 상태 확보]
        │
        ├─► 인지/연산: 사람에게 검증·완성 비용이 큼
        ├─► 탐색: 유효 배치 도달 경로·재현 절차 부재
        ├─► 기준: 다중 해 vs 단일 정답 정의 모호
        ├─► 과정: 오류 전파·데드락, 되돌림 필요
        └─► 맥락: 학습/데모/퍼즐 목적 혼재 → 기능 요구 충돌
```

**한 줄**: "완성해야 한다"는 동기는 분명하지만, **수동으로는 비용이 크고, 성공 기준·도달 방법·프로그램 역할**이 엮여 있어 불편함이 **개인 실수가 아니라 문제 구조**에서 나온다.

## STEP 3 — Why #2

### 질문

**왜 단순 계산이 아니라 프로그램으로 구현하는가?**

### 1. 반복 가능성 (Repeatability)

| 측면 | 단순 계산 | 프로그램 |
|------|-----------|----------|
| 동일 입력 → 동일 결과 | 보장 어려움 | 동일 규칙이면 재현 가능 |
| 다른 시작 조건 | 매번 처음부터 | 입력·규칙만 바꿔 반복 |
| 공유 | 과정 생략 | 코드·실행으로 과정까지 공유 |
| 확장 | 규모 변경 시 방법 전면 수정 | 구조 유지하며 확장 |

> 프로그램은 "한 번의 성공"을 **재현·반복·변형 가능한 절차**로 바꾼다.

### 2. 검증 자동화 (Automated Verification)

- 10개 선 + 중복 없음 = **규칙이 분명한 판정 문제**
- 일관된 기준, 즉시 피드백, 부분 검증(조기 실패 감지) 가능
- 사람의 "대충 맞는 것 같다"를 **동일한 판정**으로 대체

### 3. 오류 방지 (Error Prevention)

| 오류 유형 | 예 |
|-----------|-----|
| 산술 | 합 계산 착오 |
| 제약 위반 | 같은 숫자 두 번 사용 |
| 구조적 | 앞 칸 배치로 뒤가 불가능 |
| 기준 혼동 | 유효한 다른 해를 오답 처리 |
| 절차 누락 | 대각선 검증 생략 |

프로그램은 **발생 전 차단 + 발생 후 국소화**의 층을 제공한다.

### 4. 규칙 기반 사고 훈련 (Rule-based Thinking Training)

| 사고 요소 | 4×4 마방진 대응 |
|-----------|-----------------|
| 불변 조건 | 모든 선 합 = 34, 숫자 집합 = {1..16} |
| 생성 규칙 | 4×4 구성 패턴 |
| 검증 규칙 | 10개 선 순회 |
| 데이터 구조 | 2D 배열, 행·열·대각 인덱싱 |
| 관심사 분리 | 생성 vs 검증 vs 출력 |

> "계산"은 한 번의 답을, "프로그램"은 **규칙을 명시·실행·검증 가능한 형태**로 만든다.

### 네 관점 통합

```
[Why #1: 유효한 4×4 배치를 확보·확인·학습·공유]
        │
        ├─► 반복 가능성
        ├─► 검증 자동화
        ├─► 오류 방지
        └─► 규칙 기반 사고 훈련
                │
                ▼
단순 계산 = 일회성·수동·암묵적 규칙
프로그램   = 재현·자동·명시적 규칙
```

## STEP 4 — Why #3 (핵심 발견)

### 질문

**왜 이 문제를 TDD 방식으로 설계하려 하는가?**

TDD는 "테스트를 먼저 쓰는 코딩 기법"이 아니라, **무엇이 통제되어야 하고, 무엇이 절대 깨지면 안 되며, 무엇을 주고받아야 하는지**를 먼저 고정하는 **설계 방식**이다.

### 1. 무엇이 통제되어야 하는가?

**통제해야 할 것**

| 통제 대상 | 이유 | TDD 역할 |
|-----------|------|----------|
| 10개 선의 합 | "완성"의 판정 기준 | 핵심 assertion |
| 숫자 집합 {1..16} | 중복·누락 = 구조적 실패 | 선행 조건 |
| 마법常数 34 | 도메인에서 유도되는 고정값 | 기대값 고정 |
| 판정 결과 valid/invalid | "완성 ≠ 정답" 혼선 방지 | 단일 판정 계약 |
| 생성 규칙의 결정성 | 반복 가능성 | 동일 입력 → 동일 출력 |
| 실패 지점 | 오류 전파·데드락 | 통제된 출력으로 분리 |

**통제하지 않아도 되는 것**

| 비통제 대상 | 이유 |
|-------------|------|
| 특정 하나의 배치 형태 | 다중 해 존재 |
| 생성 방법의 내부 순서 | 출력이 불변 만족하면 교체 가능 |
| UI·출력 형식 | 도메인 검증과 분리 가능 |

### 2. 어떤 불변 조건이 존재하는가?

**도메인 불변 (I1~I5)**

```
I1. ∀ 행 r: sum(grid[r][*]) = M
I2. ∀ 열 c: sum(grid[*][c]) = M
I3. sum(주대각) = M, sum(부대각) = M
I4. grid 값 집합 = {1, 2, …, 16} (정확히 한 번씩)
I5. M = 34
```

**연산 불변**

| 연산 | 불변 |
|------|------|
| 검증 | I1~I4 만족 → true, 하나라도 위반 → false |
| 생성 | 출력은 항상 I1~I4 만족 |
| 부분 상태 | 채운 칸 + 사용 숫자가 I4 부분 버전 위반하지 않음 |

**설계 불변 (I6~I8)**

```
I6. 검증 로직은 생성 로직과 분리
I7. "유효" 정의는 회전·대칭 변형에도 동일 적용
I8. 테스트는 구현 세부가 아니라 I1~I4 기준으로 통과/실패 선언
```

**핵심 발견**: 마방진은 탐색 문제처럼 보이지만 **본질은 불변 조건 만족 문제**이다.

### 3. 왜 입력/출력이 명확해야 하는가?

**입력이 명확해야 하는 이유**

- 크기·범위·중복 규칙 불명 → 테스트 케이스 작성 불가
- 빈 칸 허용 여부 불명 → valid/false 기준 충돌
- 1~16이 아닌 집합 → M=34 불변과 충돌

**출력이 명확해야 하는 이유**

- "대충 맞는 것 같음" → 완성 ≠ 정답 혼선 재발
- valid만 반환 → 어디가 틀렸는지 학습·디버깅 불가
- "한 가지 정답"만 출력 → 다중 해 오답 처리

**입력/출력 명확성 ↔ TDD 연결**

```
명확한 I/O 계약 → 테스트 작성 가능 → 불변 I1~I4 실행 전 고정
    → 구현 교체 가능 → 반복·자동검증·오류 방지 달성
```

### Why #3 통합

> **4×4 마방진 프로그램의 본질은 "16칸을 채우는 UI"가 아니라, I1~I4 불변 조건을 통제·검증·재현하는 계약(contract)이다. TDD는 그 계약을 코드보다 먼저 쓰는 설계 방식이다.**

## STEP 5 — 진짜 문제 정의

### 1. 표면 문제 정의 (잘못된 정의)

> **「1부터 16까지 숫자를 4×4 칸에 넣어서, 가로·세로·대각선 합이 모두 34가 되게 만드는 프로그램을 만든다.」**

| 한계 | 설명 |
|------|------|
| 행위 중심 | "만든다"만 — 확인·보장·재현 없음 |
| 판정 기준 누락 | 16칸 채움 = 완료인지 불명 |
| 책임 범위 불명 | 생성 / 검증 / 시연 혼재 |
| 단일 정답 가정 | 다중 유효 배치 무시 |
| 도구 목적 상실 | 프로그램이 왜 필요한지 없음 |

### 2. 개선된 문제 정의 (정확한 정의)

> **「1부터 16까지의 서로 다른 정수를 4×4 격자에 배치했을 때, 네 행·네 열·두 대각선 — 총 열 개의 선 — 각각의 합이 동일한지를 일관되게 판정하고, 그 조건을 만족하는 배치를 재현 가능하게 확보할 수 있는 도구를 만든다. 이 도구는 사람의 수동 계산·기억·눈대중 검증에 의존하지 않고, 동일한 기준으로 반복 실행·결과 공유·실패 원인 파악이 가능해야 한다.」**

| 요소 | 내용 |
|------|------|
| **대상** | 4×4 격자, {1..16}, 열 개의 선 |
| **핵심 과업** | (1) 조건 만족 판정, (2) 조건 만족 배치 확보 |
| **품질 요구** | 일관된 판정, 재현 가능, 실패 시 근거 파악 |
| **명시적 배제** | 특정 하나의 모양만 정답으로 인정하지 않음 |

**문제 경계**

- **In**: 유효성 판정, 유효 배치 확보, 동일 기준 반복 실행
- **Out (가능)**: UI, 난이도, 퍼즐 게임화, n×n 일반화

### 3. 핵심 Invariant (STEP 5 확정본)

**구조 Invariant — 유효한 4×4 배치**

| ID | 불변 조건 |
|----|-----------|
| I1 | 각 행의 합이 모두 같다 |
| I2 | 각 열의 합이 모두 같다 |
| I3 | 두 대각선의 합이 그 목표값과 같다 |
| I4 | 1~16 각 숫자가 정확히 한 번씩 |
| I5 | 공통 목표값 = 34 |

**도구 Invariant**

| ID | 불변 조건 |
|----|-----------|
| I6 | 판정 기준은 I1~I5에만 근거 |
| I7 | "유효" 판정 → I1~I5 모두 만족 |
| I8 | "무효" 판정 → I1~I5 중 하나 이상 위반 |
| I9 | 배치 확보 결과 → I1~I5 만족 |
| I10 | 동일 입력·규칙 → 재현 가능 |

**설계 Invariant**

| ID | 불변 조건 |
|----|-----------|
| I11 | "조건 만족?"과 "어떻게 얻었는가?" 분리 |
| I12 | 회전·대칭 변형도 I1~I5 만족 시 동등하게 유효 |

### 4. 훈련하려는 사고 능력

| # | 능력 | 내용 |
|---|------|------|
| 1 | **제약 동시 만족** | 열 개 선에 동시 영향; 부분 성공 vs 전체 조건 |
| 2 | **불변 조건 명시화** | "맞다"를 I1~I5 진술로 표현 |
| 3 | **I/O·판정 계약 설계** | 채움 / 유효 / 실패 구분; 검증 가능한 진술 |
| 4 | **관심사 분리** | 판정 vs 확보; 도메인 vs 표현 |
| 5 | **실패 구조화** | 어떤 불변 위반인지; 진단 가능한 실패 |
| 6 | **재현·공유 인식** | 반복 가능한 절차; 판정 기준 공유 |

## STEP 6 — 첫 번째 검증 가능한 요구

### 목적

"무엇부터 맞아야 하는가" 순서 고정, 불변 연결, 첫 TDD 사이클 범위 제한.

### 우선순위 원칙

```
도메인 정의 → 판정(validate) → 확보(generate) → 부가 정보
```

**판정이 먼저** — I11(획득 방식 독립), I6~I8(신뢰의 출발점), I9 검증에 선행 조건.

### 첫 번째 검증 질문

> **「알려진 유효 배치 하나에 대해, 판정 결과는 '유효'인가?」**

| 기준 | 설명 |
|------|------|
| 가장 작은 성공 케이스 | "맞는 것 하나 인정"이 기준선 |
| I1~I5 직접 연결 | 유효 = I1~I5 동시 만족 |
| I11 준수 | 획득·생성과 무관 |
| 이후 테스트의 심판 | 모든 후속 테스트가 이 판정 재사용 |

**첫 질문이 아닌 것**: 배치 생성, 34 단독, 특정 모양 일치, UI.

### 테스트 질문 로드맵

#### Phase A — 판정 계약 (최우선)

| 순서 | 검증 질문 | 불변 |
|------|-----------|------|
| A1 | 알려진 유효 배치 → 유효 | I1~I5, I7 |
| A2 | 숫자 중복 → 무효 | I4, I8 |
| A3 | 1~16 밖 값 → 무효 | I4, I8 |
| A4 | 한 행 합 불일치 → 무효 | I1, I8 |
| A5 | 한 열 합 불일치 → 무효 | I2, I8 |
| A6 | 대각선 하나 불일치 → 무효 | I3, I8 |
| A7 | 다른 유효 배치도 동일하게 유효 | I6, I12 |

#### Phase B — 도메인 상수

| 순서 | 검증 질문 | 불변 |
|------|-----------|------|
| B1 | 목표 합 = 34 | I5 |
| B2 | 목표값이 도메인에서 유도됨 | I5, I6 |

#### Phase C — 확보 계약 (A·B 이후)

| 순서 | 검증 질문 | 불변 |
|------|-----------|------|
| C1 | 확보 결과 → 판정 통과 | I9, I7 |
| C2 | 동일 규칙·조건 → 동일 결과 | I10 |
| C3 | (선택) 확보 실패 시 반쪽 유효 금지 | I9 |

#### Phase D — 실패 구조화 (선택)

| 순서 | 검증 질문 | 불변 |
|------|-----------|------|
| D1 | 무효 시 위반 종류 구분 | 실패 구조화 |
| D2 | 무효 판정과 위반 정보 모순 없음 | I8 |

### 첫 사이클 범위

**In**

- 판정 A1~A7 (최소 A1~A4부터)
- 도메인 상수 B1
- 확보 C1

**Out**

- 사용자 퍼즐 UI
- 3×3 / 5×5 일반화
- 회전·대칭 생성
- 실패 상세 D1~D2 (최소 계약 이후)
- 화면·입력 방식

### STEP 5 → STEP 6 수렴

```
[진짜 문제] 유효 배치를 일관되게 판정하고, 재현 가능하게 확보
              │
              ▼
[첫 질문] 알려진 유효 배치 → 유효 (A1)
              │
              ├─► A2~A7 : 무효·다중해
              ├─► B1     : I5 (34)
              └─► C1     : I9 (확보)
```
