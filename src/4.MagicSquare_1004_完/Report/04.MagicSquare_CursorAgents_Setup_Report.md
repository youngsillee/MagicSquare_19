## 1. 개요

MagicSquare TDD 연습 프로젝트에서 역할별 Cursor Custom Agent 8종을 설계하고 `.cursor/agents/`에 저장했다. 각 Agent는 ECB 아키텍처, Dual-Track TDD(Logic Track + UI Contract Track), RED-GREEN-REFACTOR 사이클, 공통 안전 규칙을 공유하며 역할별 Responsibilities·Workflow·Must Not·Output Format만 분화한다.

이후 `backup-report-github-manager` Agent에 **작업 보고서(Report/)** 와 **프롬프트 원문 로그(Prompting/)** 작성 책임을 추가하여, 세션 산출물을 쌍(pair)으로 문서화할 수 있게 했다.

## 2. Agent 설계 결정

### 2.1 저장 위치·파일 규칙

| 항목 | 결정 |
|------|------|
| 저장 경로 | `.cursor/agents/<agent-name>.md` |
| 파일 수 | Agent 1개 = Markdown 1개 |
| 파일명 | Agent 이름과 동일 (kebab-case) |
| SSOT 참조 | `.cursorrules`, `Report/01`, `Report/02` |

### 2.2 역할 분리 원칙

| 원칙 | 내용 |
|------|------|
| Track 분리 | backend-developer = Logic Track, frontend-developer = UI Contract Track |
| 구현 vs 기획 | product-planning-manager는 계획·명세만, 코드 미작성 |
| 품질 게이트 | quality-assurance-engineer는 TDD·ECB 위반 탐지·차단 |
| 문서화 | backup-report-github-manager는 Report + Transcript 쌍 Export |
| AI 가드레일 | ai-integration-expert는 Agent/rules/MCP 정렬 |

### 2.3 공통 Must Not (전 Agent)

- 사용자 승인 없이 파일 삭제, Git push, 배포, DB 변경 금지
- secret 출력·커밋 금지
- RED 확인 없이 production 코드 작성 금지
- 테스트 약화·삭제·우회 금지
- ECB 경계 위반 금지 (entity → boundary/control import 등)
- 타입 힌트 없는 public 함수, print() 디버깅, PEP8 위반 금지

## 4. 공통 프롬프트 구조

모든 Agent Markdown은 아래 8개 섹션을 동일 순서로 포함한다.

```
# 저장 경로
# Agent Name
# Role
# Responsibilities
# Workflow
# Must Not
# Output Format
```

### 4.1 MagicSquare 도메인 규칙 (Agent 공유)

| 항목 | 규칙 |
|------|------|
| 입력 | 4×4 int 행렬, 0=빈칸(정확히 2개), 값 0 또는 1~16, 0 제외 중복 금지 |
| 출력 | int[6] = [r1,c1,n1,r2,c2,n2], 좌표 1-index |
| MagicConstant | 34 |
| 해 결정 | (작은수→첫빈칸, 큰수→둘째빈칸) 우선 → 실패 시 반대 조합 |

### 4.2 Dual-Track TDD 제약 (Agent 공유)

| Track | 테스트 위치 | Mock |
|-------|-------------|------|
| Logic Track | `tests/entity/` | Domain Mock 금지 |
| UI Contract Track | `tests/boundary/` | Control/Entity Mock 허용 |
