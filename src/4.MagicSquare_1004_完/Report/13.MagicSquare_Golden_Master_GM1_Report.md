## 3. Golden Master 설계

### 3.1 캡처 전략

| 방식 | 채택 | 사유 |
|------|------|------|
| stdout capture | ❌ | Boundary 계약은 DTO envelope (`SuccessResponse` / `FailureResponse`) |
| **Result DTO serialize** | ✅ | PRD §12.2·§12.3과 1:1 대응, `.cursorrules` print 금지 준수 |

### 3.2 직렬화 규칙

| 결과 | 형식 |
|------|------|
| 성공 | `Output:\n[r1,c1,n1,r2,c2,n2]` (쉼표 구분, 공백 없음) |
| 검증 실패 | `Error:\n{code}` (E002, E005 등) |
| Domain 무해 | `Error:\nE006` (`UnsolvableDomainError` → PRD §12.3 계약 코드) |

> **참고:** 현재 `UIBoundary`는 G3 입력 시 `UnsolvableDomainError`를 **예외로 전파**한다. Golden Master 캡처 레이어에서 E006으로 직렬화하여 ErrorMapper 구현 후에도 기대값이 안정적으로 유지되도록 설계했다.

### 3.3 기준 파일 구조

- Section 헤더: `[scenario_name]`
- 본문: `Input:` (4×4 격자) + `Output:` 또는 `Error:`
- 구분선: `________________________________________` (40 underscores)
- SSOT: `tests/golden_master_expected.txt`

## 10. 알려진 정합 이슈

### ISS-013-01 — 예시 vs 실제 에러 코드

| 출처 | `invalid_blank_count` 코드 |
|------|---------------------------|
| GM 프롬프트 예시 | `INVALID_BLANK_COUNT` |
| 실제 Boundary (`InputValidator`) | `E002` |
| Golden Master 기준 | **`E002`** (실제 출력 SSOT) |

### ISS-013-02 — E006 vs UIBoundary 예외 (Report/12 연계)

G3 입력 시 `UIBoundary.solve()`는 `UnsolvableDomainError`를 **아직** `FailureResponse(E006)`으로 매핑하지 않는다. Golden Master는 E006으로 직렬화 — U-OUT-03 GREEN 후에도 기대값 유지.

### ISS-013-03 — GM-04 테스트 파일명

README GM-04 라벨: `test_golden_master_magic_square`  
실제 pytest node: `tests/test_gm_01_magic_square_golden_master.py`
