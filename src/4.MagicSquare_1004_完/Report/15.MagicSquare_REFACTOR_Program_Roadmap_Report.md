## 1. 개요

본 세션은 Report/14 REFACTOR 준비 분석 이후, **Dual-Track Safe Refactoring 프로그램**을 실행 가능한 형태로 구체화했다. 초기 범용 REFACTOR 프롬프트를 평가하고 **단일 커밋(Prompt A)** 과 **전체 로드맵(Prompt B)** 2종으로 분리했으며, Prompt A 실행 시 Step 0 게이트 미충족으로 REFACTOR를 중단·Phase 0 선행 작업을 정의했다. Prompt B Ask-mode 분석으로 Wave 1 착수 조건·C1 계획·ISS-012-01 GM 영향을 확정했다.

| Phase | Step | 내용 | 산출 |
|-------|------|------|------|
| Ask | Turn 1 | REFACTOR 프롬프트 평가 + Prompt A/B 2종 설계 | 프롬프트 SSOT |
| Agent | Turn 2 | Prompt A Step 0 — pytest/Gate 확인 | REFACTOR **중단** (RED 19) |
| Ask | Turn 3 | Prompt B — Phase 0·Wave 1·C1·ISS-012-01 | 본 Report |
| Export | Turn 4 | Report/15 + Prompting/15 | ✅ |

## 3. 프롬프트 평가 및 2종 분리

### 3.1 원본·보완 프롬프트 강점

| 영역 | 평가 |
|------|------|
| 계약 보호 | Input/Output, Error Contract, row-major, 1-index, int[6] 고정 |
| Dual-Track | Boundary(UI) vs Domain(Logic) ECB 정합 |
| TDD 절차 | 테스트 → 리팩토링 → 재테스트, GREEN 유지 |
| Change Budget (보완판) | 커밋 1개 제한 — AI 과잉 리팩토링 방지 |
| Golden Master (보완판) | GM-1 회귀 앵커 명시 — Report/13 연계 |

### 3.2 약점 및 보완

| 문제 | 보완 (Prompt A/B 반영) |
|------|------------------------|
| `domain.py`/`boundary.py` 가상 경로 | `@src/control/`, `@src/boundary/` 실경로 + ECB 레이어 |
| RED 19건 전제 누락 | Phase 0 게이트 G-01~G-05 명시 |
| R-L/R-U vs Report/14 불일치 | RF-01~08 ID로 Report 순번 1~11 매핑 |
| GM 검증 명령 미구체화 | 고정 명령 블록 삽입 |
| 단일·전체 역할 혼재 | **Prompt A** (1커밋) / **Prompt B** (다회차 orchestration) 분리 |

### 3.3 Prompt A — 단일 커밋 Safe Refactor

- **용도:** Agent mode, 커밋 1개, Change Budget 엄격
- **전제:** pytest 전체 GREEN + GM-1 matched
- **절차:** Step 0~5 (기준선 → 목표 1개 → 보호 테스트 → Refactor → 회귀 → 산출물)

### 3.4 Prompt B — REFACTOR 프로그램 (다중 커밋)

- **용도:** Ask mode 로드맵 + Agent mode 다회차 (매번 Prompt A 호출)
- **Wave 1~4:** P0 계약·SSOT → P1 DRY·Screen → P2 Entity → P2 Boundary polish
- **금지:** Wave 1 미완료 시 R-L* 선행, 한 세션 Wave 전체 구현

| D-SOL-01 (GREEN) | `[2,2,10,3,3,7]` |
| SC-CTL-001 (GREEN) | `[2,2,7,3,3,10]` |

### C3 RF-03 시나리오

- GM `reverse_success` → **diff**
- SC-CTL-001, U-OUT-01 → **FAIL**

### 권장 (옵션 A)

PRD `[r1,c1,n1,r2,c2,n2]`에서 n1/n2 = **실제 배치 값** → SSOT = Solver physical `[2,2,10,3,3,7]`

| 단계 | 작업 |

| C3 후 | GM diff 확인 → Report 근거 → `--approve-golden` |
