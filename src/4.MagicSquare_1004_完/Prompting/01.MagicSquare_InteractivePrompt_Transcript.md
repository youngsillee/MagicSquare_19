# 4×4 Magic Square — 대화형 프롬프트 Transcript

| 항목 | 내용 |
|------|------|
| **프로젝트** | MagicSquare_1004 |
| **용도** | 문제 정의 워크플로(STEP 1~6)를 대화형으로 재현·재개 |
| **형식** | 역할 설정 → 단계별 User Prompt → (선택) Export Prompt |
| **작성일** | 2026-05-23 |

---

## 사용 방법

1. **「0. 역할 설정」** 을 AI에게 먼저 전달한다.
2. **STEP 1**부터 순서대로 User Prompt를 한 단계씩 전달한다.
3. 각 STEP 응답을 확인한 뒤 다음 STEP으로 진행한다.
4. STEP 6 완료 후 **「Export — Report」** 또는 **「Export — Transcript」** 로 산출물을 내보낸다.
5. 구현·설계 단계는 별도 프롬프트 세트로 분리한다. (본 transcript 범위 밖)

---

## 0. 역할 설정 (System / Opening Prompt)

```
당신은 문제 정의 전문가입니다.
나는 4x4 Magic Square 프로그램을 만들려고 합니다.
하지만 바로 설계하거나 구현하지 마십시오.
먼저 문제 인식 단계부터 진행하십시오.

규칙:
- 구현 설계는 하지 마십시오 (STEP 5까지).
- STEP 5에서는 코드 언급, 알고리즘 설명을 하지 마십시오.
- 각 STEP은 지시된 질문·관점에만 답하십시오.
- "마방진을 만든다"는 표현 대신, 관찰·제약·판정 관점으로 서술하십시오.
- 응답은 구조화된 Markdown으로 작성하십시오.
- 사용자가 "진행" 또는 "네 진행"이라고 하면, 합의된 다음 STEP으로 이어가십시오.
```

---

## STEP 1 — Observation (관찰)

### User Prompt

```
--------------------------------------------
STEP 1 — Observation (관찰)
--------------------------------------------
다음 질문에 답하십시오:
1. 현재 우리가 해결하려는 상황은 무엇인가?
2. 왜 4x4 마방진 문제를 다루는가?
3. 이 문제는 어떤 학습 또는 시스템 설계 맥락에서 등장하는가?
"마방진을 만든다"는 표현 대신, 상황을 관찰 관점으로 서술하십시오.
```

### 기대 응답 구조 (가이드)

- 3개 질문 각각에 대한 답
- 표 또는 목록으로 관찰 요소 정리 (입력 / 구조 / 제약 / 미지)
- 관찰 관점 종합 (한 문장)

### 완료 조건

- [ ] "16칸 채우기"가 아닌 "10개 선의 합 일치 제약"으로 서술됨
- [ ] 4×4 규모 선택 이유가 명시됨
- [ ] 학습·설계 맥락이 분리되어 서술됨

---

## STEP 2 — Why #1

### User Prompt

```
--------------------------------------------
STEP 2 — Why #1
--------------------------------------------
Q: 왜 마방진을 완성해야 하는가?
A를 가정하여 답하고, 그 답에서 드러나는 불편함 또는 구조적 문제를 분석하십시오.
```

### 기대 응답 구조 (가이드)

- **가정 (A)** 한 문장 명시
- 가정에서 드러나는 **불편함** (인지 부하, 검증 비용, 탐색, 다중 해, 오류 전파, 목적 혼선 등)
- **구조적 문제** 요약 (개인 실수 vs 문제 구조)

### 완료 조건

- [ ] "완성"의 목적이 가정으로 명시됨
- [ ] 불편함이 3가지 이상 구조적으로 분석됨
- [ ] 다음 STEP(TDD·프로그램 필요성)으로 연결 가능

---

## STEP 3 — Why #2

### User Prompt

```
--------------------------------------------
STEP 3 — Why #2
--------------------------------------------
Q: 왜 단순 계산이 아니라 프로그램으로 구현하는가?
다음 관점에서 답하십시오:
- 반복 가능성
- 검증 자동화
- 오류 방지
- 규칙 기반 사고 훈련
```

### 기대 응답 구조 (가이드)

- 4개 관점 각각: 단순 계산 한계 → 프로그램이 주는 것
- 4개 관점 **통합** 다이어그램 또는 표
- STEP 2 구조적 문제와의 연결

### 완료 조건

- [ ] 4개 관점 모두 다룸
- [ ] "계산 vs 프로그램" 대비가 명확함
- [ ] TDD(STEP 4)로 자연스럽게 이어짐

---

## STEP 4 — Why #3 (핵심 발견)

### User Prompt

```
--------------------------------------------
STEP 4 — Why #3 (핵심 발견)
--------------------------------------------
Q: 왜 이 문제를 TDD 방식으로 설계하려 하는가?
- 무엇이 통제되어야 하는가?
- 어떤 불변 조건이 존재하는가?
- 왜 입력/출력이 명확해야 하는가?
```

### 기대 응답 구조 (가이드)

- **통제 대상** vs **비통제 대상** 표
- **불변 조건** (I1~I5 도메인, 연산·설계 불변)
- **입력/출력 명확성** — 모호할 때의 문제, 명확할 때의 이점
- **Why #3 통합** — "본질은 불변 조건 만족 + 계약(contract)"

### 완료 조건

- [ ] 통제/비통제 구분됨
- [ ] I1~I5 수준의 불변이 명시됨
- [ ] I/O 계약과 TDD 연결이 설명됨

---

## STEP 5 — 진짜 문제 정의

### User Prompt

```
--------------------------------------------
STEP 5 — 진짜 문제 정의
--------------------------------------------
다음을 작성하십시오:
1. 표면 문제 정의 (잘못된 정의)
2. 개선된 문제 정의 (정확한 정의)
3. 이 문제의 핵심 Invariant
4. 우리가 실제로 훈련하려는 사고 능력

− 구현 설계는 하지 마십시오.
− 코드 언급 금지.
− 알고리즘 설명 금지.
− 구조화된 Markdown으로 작성하십시오.
```

### 기대 응답 구조 (가이드)

1. 표면 정의 + **왜 잘못되었는지**
2. 개선 정의 + **In/Out 경계**
3. Invariant 표 (I1~I12 또는 동등 구조)
4. 사고 능력 4~6항 (제약 동시 만족, 불변 명시, I/O 계약, 관심사 분리, 실패 구조화, 재현성)
5. **요약** 표

### 완료 조건

- [ ] 코드·알고리즘·구현 설계 없음
- [ ] 표면 vs 진짜 문제 대비 명확
- [ ] Invariant가 검증 가능한 진술 형태
- [ ] STEP 6(첫 테스트 질문)으로 이어질 수 있음

---

## STEP 6 — 첫 번째 검증 가능한 요구

### User Prompt (대화 연속 — "네 진행")

```
네 진행
```

또는 명시적으로:

```
STEP 6 — 첫 번째 검증 가능한 요구

STEP 5의 문제 정의·불변·훈련 목표를 바탕으로,
무엇을 먼저 "검증 가능한 질문"으로 고정할지 정리하십시오.

포함:
- 첫 번째 검증 질문 (The First Test Question)과 선택 이유
- Phase A~D 테스트 질문 로드맵
- 첫 TDD 사이클 In/Out 범위
- STEP 5 훈련 목표와의 매핑

구현·코드·알고리즘은 다루지 마십시오.
```

### 기대 응답 구조 (가이드)

- **우선순위 원칙**: 판정 → 확보
- **첫 질문**: "알려진 유효 배치 → 유효"
- **Phase A~D** 로드맵 표
- **In/Out** 범위
- **STEP 7 예고** (입력·출력 계약, 기준 배치)

### 완료 조건

- [ ] 첫 질문이 A1(판정)으로 고정됨
- [ ] 확보(C)가 판정(A) 이후로 순서 명시됨
- [ ] 구현 없이 검증 질문만 정의됨

---

## STEP 7 — (예고) 입력·출력 계약

> STEP 6 완료 후 사용. 본 transcript 원본 대화에서는 "다음 단계 제안"으로만 언급됨.

### User Prompt

```
--------------------------------------------
STEP 7 — 입력·출력 계약
--------------------------------------------
STEP 5~6을 바탕으로 다음만 문장 수준으로 고정하십시오:

1. 입력 계약 — 격자 크기, 값 범위, 빈 칸 허용 여부, 타입
2. 출력 계약 — valid/invalid, (선택) 실패 정보 포함 여부
3. A1용 기준 배치 — 문서에 고정할 "알려진 유효 배치" 하나

구현·코드·알고리즘은 하지 마십시오.
```

## Export Prompts

### Export — Report

```
지금까지 작업한 내용을 Report 폴더에 보고서로 내보내줘
```

**의도**: STEP 1~6 산출물을 `Report/` 폴더에 통합 Markdown 보고서로 저장.

---

### Export — Transcript (본 문서)

```
현재까지의 프롬프트를 대화형 프롬프트로 Prompting 폴더에 Export transcript 해줘
```

**의도**: 재사용 가능한 단계별 User Prompt·가이드·완료 조건을 `Prompting/` 폴더에 저장.

## 파일 관계

```
MagicSquare_1004/
├── Report/
│   └── MagicSquare_ProblemDefinition_Report.md   ← STEP 1~6 산출물 (답변 통합)
```
