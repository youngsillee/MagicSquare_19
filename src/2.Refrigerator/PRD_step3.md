# PRD — 3단계: 사용자 프로필 및 레시피 저장

## 1. 문서 정보

| 항목 | 내용 |
|------|------|
| 제품 | 냉장고 사진 기반 재료 인식 · 레시피 추천 웹앱 |
| 단계 | 3단계 — 사용자 프로필, 레시피 영구 저장 |
| 버전 | 0.1 (초안) |
| 선행 조건 | **1단계**(재료 인식), **2단계**(레시피 생성) 플로우 및 데이터 스키마 확정 |

---

## 2. 목표

**로그인한 사용자**에 대해 프로필을 유지하고, **2단계에서 생성된 레시피**를 **저장·목록 조회·삭제(소프트 삭제 선택)**할 수 있게 한다. 선택적으로 **1단계 인식 결과 스냅샷**도 함께 묶어 “어떤 사진/재료에서 나온 레시피인지” 추적 가능하게 한다.

### 2.1 성공 기준 (3단계 완료 정의)

- 회원가입/로그인/로그아웃이 동작한다(구체 인증 방식은 아래 “기술 선택”에서 결정).
- 로그인 사용자만 자신의 레시피를 생성·조회·삭제할 수 있다(다른 사용자 데이터 접근 불가).
- 저장된 레시피는 **제목 검색 또는 태그(선택)**로 목록에서 찾기 쉽다(최소: 시간순 정렬).
- 비로그인 사용자는 1·2단계 체험은 가능하되, **저장 시 로그인 유도**(제품 정책으로 확정).

---

## 3. 사용자 스토리

1. **사용자**로서 계정을 만들고 **만든 레시피를 보관**하고 싶다.
2. **사용자**로서 **저장한 레시피 목록**을 보고, 하나를 열어 전체 내용을 다시 보고 싶다.
3. **사용자**로서 잘못 저장한 레시피를 **삭제**하고 싶다.
4. **사용자**로서 **닉네임·식이 선호** 같은 프로필을 수정해 다음 생성에 반영하고 싶다(선택).

---

## 4. 범위 (In Scope)

### 4.1 인증·프로필

- 이메일/비밀번호 또는 OAuth 중 **1가지 이상** (PRD 레벨에서 확정; 구현은 별도 기술 스파이크).
- 프로필 필드(최소): `user_id`, `display_name`, `created_at`, `updated_at`.
- (선택) `dietary_notes`, `default_servings` — 2단계 기본값으로 전달.

### 4.2 레시피 저장

- 엔티티: **SavedRecipe** (또는 동등 명칭).
- 필수 연관: `owner_user_id`, `payload`(2단계 JSON), `title`(검색용 중복 저장 허용), `created_at`.
- (선택) `source_ingredients_snapshot`(1단계 JSON), `thumbnail_url`(후속), `tags[]`.

### 4.3 API (예시 수준)

- `POST /api/recipes` — 저장(본문에 2단계 결과 + 메타).
- `GET /api/recipes` — 내 목록(페이지네이션).
- `GET /api/recipes/:id` — 단건.
- `DELETE /api/recipes/:id` — 삭제.

### 4.4 UI

- 로그인/회원가입 화면, 프로필 편집(최소 닉네임).
- “레시피 저장” 버튼(2단계 결과 화면), “내 레시피” 목록·상세.

---

## 5. 비범위 (Out of Scope) — 3단계에서 하지 않음

- 소셜 공유, 댓글, 평점 커뮤니티.
- 팀/가족 공유 캘린더, 식단 플래너.
- 결제·구독(후속 상용화 시 별도 PRD).

## 6. 기능 요구사항

### 6.1 인증·보안

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F3.1 | 세션/JWT 등 표준 패턴, HTTPS 배포 전제 | P0 |
| F3.2 | 비밀번호 해시(bcrypt/argon2 등), 레이트 리밋(로그인) | P0 |
| F3.3 | 모든 레시피 API는 소유자 검증 | P0 |
| F3.4 | OpenRouter API 키는 계속 서버 전용(변경 없음) | P0 |

### 6.2 데이터

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F3.5 | RDB 또는 BaaS(예: Supabase) 중 하나로 영구 저장 | P0 |
| F3.6 | 마이그레이션/스키마 버전 관리 | P1 |
| F3.7 | (선택) 삭제는 `deleted_at` 소프트 삭제 | P2 |

### 6.3 제품 정책

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F3.8 | 비로그인 시 저장 버튼 → 로그인 모달/리다이렉트 | P0 |
| F3.9 | 계정 삭제 시 연관 레시피 처리 정책(함께 삭제 권장) 문서화 | P1 |

## 7. 비기능 요구사항

- **개인정보**: 이메일 최소 수집, 약관·개인정보 처리방침 링크(배포 시 필수).
- **가용성**: DB 백업 주기(운영 문서).
- **성능**: 목록 API p95 목표(예: 500ms 내) — 인프라에 맞게 조정.

---

## 8. 데이터 모델 (초안)

**User**

- `id`, `email`(또는 OAuth subject), `password_hash`(해당 시), `display_name`, timestamps

**SavedRecipe**

- `id`, `user_id` (FK)
- `title`, `body` (JSONB 또는 TEXT) — 2단계 스키마 그대로 저장 권장
- `ingredients_snapshot` (JSONB, 선택)
- `vision_model`, `recipe_model` (문자열, 선택) — 재현/디버깅
- `created_at`, `updated_at`, `deleted_at`(선택)

---

## 9. 1·2단계와의 연계

- **1단계**: (선택) 업로드 세션 ID를 두고, 저장 시 스냅샷에 포함.
- **2단계**: `POST /api/recipes`의 본문은 2단계 출력 스키마를 채택; 서버가 `user_id`와 결합.

---

## 10. 리스크 및 완화

| 리스크 | 완화 |
|--------|------|
| 인증 구현 복잡도 | BaaS 또는 검증된 라이브러리 사용 |
| JSON 페이로드 크기 | DB 타입/압축, steps 길이 제한(2단계와 협의) |
| GDPR/삭제 요청 | 계정 삭제 플로우 + 데이터 삭제 |

---

## 11. 인수 테스트 (예시)

1. 사용자 A가 레시피 저장 → 목록에 표시.
2. 사용자 B가 A의 `recipe_id`로 직접 접근 시 **403/404**.
3. 로그아웃 후 저장 시도 → 로그인 유도.
4. 계정 삭제(선택 구현 시) → 레시피 비조회.
