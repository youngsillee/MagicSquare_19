# PRD — 2단계: 재료 기반 레시피 생성 (LLM)

## 1. 문서 정보

| 항목 | 내용 |
|------|------|
| 제품 | 냉장고 사진 기반 재료 인식 · 레시피 추천 웹앱 |
| 단계 | 2단계 — 1단계 결과를 이용한 레시피 생성 |
| 버전 | 0.1 (초안) |
| 선행 조건 | **1단계** 완료: 재료 목록 API/데이터 계약 안정화 |

---

## 2. 목표

1단계에서 얻은 **재료 목록·부가 맥락**을 입력으로, **OpenRouter API**와 **`deepseek/deepseek-chat-v3.1:free`** 모델을 사용해 **지금 만들 수 있는 레시피**를 1건 이상 생성·표시한다. 사용자가 **인분, 조리 시간, 식이 제한(선택)**을 조절하면 그에 맞춰 생성 결과가 달라져야 한다.

### 2.1 성공 기준 (2단계 완료 정의)

- 1단계 산출물 없이는 레시피 생성 요청을 보낼 수 없거나, UI에서 명확히 막힌다.
- 레시피에 **제목, 인분, 재료(계량), 조리 순서**가 포함된다(형식은 아래 스키마).
- 모델이 환각한 재료를 대량 추가하는 경우를 줄이기 위한 **가드레일 프롬프트**가 있다.
- 생성 실패 시 원인 구분(네트워크, 한도, 파싱 실패) 가능한 수준의 피드백을 사용자에게 제공한다.

---

## 3. 사용자 스토리

1. **사용자**로서 인식된 재료를 보고 **“이 재료로 레시피 추천”**을 눌러 결과를 보고 싶다.
2. **사용자**로서 **2인분·20분 이내** 같은 조건을 넣고 다시 생성하고 싶다.
3. **사용자**로서 레시피가 마음에 들지 않으면 **다시 생성**할 수 있다(동일/유사 프롬프트, 시드 다양화는 선택).

---

## 4. 범위 (In Scope)

- 1단계 출력(JSON)을 서버가 그대로 또는 정규화하여 2단계 프롬프트에 주입.
- OpenRouter `chat/completions`, 모델 **`deepseek/deepseek-chat-v3.1:free`**.
- (선택) 사용자 입력: 인분, 최대 조리 시간, 주요 조리법(볶음/국/샐러드 등), 알레르기/비건 등 **텍스트만** 1단계에서 수집하지 않았다면 2단계 폼에서 수집.
- UI: 조건 폼, “생성” 버튼, 스트리밍(선택) 또는 블록 렌더링, 인쇄 친화(선택 P2).

---

## 5. 비범위 (Out of Scope) — 2단계에서 하지 않음

- 이미지 비전 호출(1단계 모델) — 단, “원본 이미지와 함께 다시 분석”은 1단계 플로우로 되돌림.
- 사용자별 **영구 저장·북마크** (**3단계**).
- 영양소 DB 연동, 구매 링크, 쇼핑 리스트 (**후속**).

---

## 6. 기능 요구사항

### 6.1 입력 결합

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F2.1 | 1단계 `ingredients[]`를 필수 입력으로 사용 | P0 |
| F2.2 | 사용자 조건(인분, 시간 상한, 식이 제한 등)을 시스템/유저 메시지에 반영 | P0 |
| F2.3 | 재료가 0개일 때 생성 버튼 비활성 또는 안내 | P0 |

### 6.2 OpenRouter 연동

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F2.4 | `OPENROUTER_API_KEY` 서버 전용 | P0 |
| F2.5 | 모델 ID: `deepseek/deepseek-chat-v3.1:free` | P0 |
| F2.6 | `max_tokens`, `temperature` 기본값 정의(재현성 vs 다양성) | P1 |
| F2.7 | 429/5xx 재시도 정책(1단계와 공통 모듈화 권장) | P1 |

### 6.3 출력 스키마 및 검증

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F2.8 | 응답을 JSON으로 강제(예: `recipes[]` 각각 `title`, `servings`, `ingredients_used`, `steps[]`, `tips`) | P0 |
| F2.9 | 스키마 검증 실패 시 1회 재생성 요청 또는 사용자에게 “형식 오류, 다시 시도” | P1 |
| F2.10 | 1단계에 없는 재료를 쓸 경우 **“추가 구매 재료”** 섹션으로 분리 표기(프롬프트로 유도) | P1 |

### 6.4 UX

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F2.11 | 생성 중 로딩·취소(선택) | P0 |
| F2.12 | “복사하기” Markdown 또는 텍스트 | P2 |

---

## 7. 비기능 요구사항

- **일관성**: 동일 입력에 대해 완전 동일 출력을 보장할 필요는 없으나, 스키마는 유지.
- **안전**: 날것 섭취·부적절 조합 등에 대한 **일반 주의 문구**(법적 면책 템플릿) UI 하단 고정 권장.
- **비용/한도**: 무료 모델 변동에 대비한 UX 카피.

---

## 8. API 계약 (2단계 산출물 → 3단계 입력)

저장 가능한 최소 단위(엔티티)를 다음으로 정의한다.

- `recipe_id`(클라이언트 임시 UUID 또는 서버 발급 — 3단계에서 영구 ID로 승격 가능)
- `title`, `servings`, `ingredients_used`, `steps`, `source_models` 메타(`vision_model`, `recipe_model`)
- `created_at`, `input_snapshot`(1단계 재료 JSON + 사용자 조건)

---

## 9. 프롬프트 가이드라인 (요약)

- 시스템: “주어진 재료 목록을 우선 소비. 부족하면 `추가 구매`로만 명시. JSON만 출력.”
- 사용자: 재료 JSON + 조건 + 원하는 레시피 개수(기본 1~3).

---

## 10. 리스크 및 완화

| 리스크 | 완화 |
|--------|------|
| 무료 모델 한도 | 재시도, 요청 간격, 토큰 상한 |
| JSON 깨짐 | 재시도, 완화 파서, 서버 측 repair 호출(선택) |
| 부적절/위험 조리 안내 | 정책 문구 + 신고/재생성 |

---

## 11. 인수 테스트 (예시)

1. 토마토·달걀·우유만 있는 시나리오 → 해당 재료 중심 레시피 + 부족 시 추가 구매 섹션.
2. 극단적으로 많은 재료 → 상위 N개만 사용하도록 프롬프트 제한 또는 요약 단계(선택).
3. API 오류 시 사용자 메시지 및 재시도 동작.
