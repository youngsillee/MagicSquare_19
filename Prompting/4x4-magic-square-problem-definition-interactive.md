# STEP 1 — Observation (관찰)

워크스페이스(`MagicSquare_xx`)에는 아직 코드나 문서가 없습니다. **“무엇을 만들지”보다 “어떤 상황을 다룰지”를 먼저 정리하는 단계**입니다.

## 1. 현재 우리가 해결하려는 상황은 무엇인가?

**4×4 격자에 1~16을 한 번씩 배치했을 때, 특정 합 조건을 만족하는 배치를 찾거나 검증·생성·탐색하려는 상황.**

- 16개의 서로 다른 정수를 4×4에 놓고, 행·열·대각선 등 **여러 선의 합이 동일한지** 판단하거나 그런 배치를 **얻어내는** 과정.
- 아직 “한 해 / 전체 해 / 검증만”은 미정. **4×4 + 1~16 + 합 일치**가 중심.
- 폴더명·대화상 **학습·실습·작은 도구** 맥락, 구현 전 **요구 선행**.

→ 해결 대상은 “마방진 앱”이 아니라 **“제약을 만족하는 4×4 정수 배치를 다루는 상황”**.

## 2. 왜 4×4 마방진 문제를 다루는가?

| 관점 | 관찰 |
|------|------|
| 규모 | 3×3보다 풍부, 5×5+보다 추적 가능 |
| 수학·퍼즐 | 규칙 있는 조합 문제의 전형 |
| 컴퓨팅 | 탐색·검증·구조화 연습에 적당한 크기 |
| 4×4 특수성 | **짝수 차수** — 홀수 차와 다른 하위 유형 |

## 3. 학습·시스템 설계 맥락

**학습**: 상태 공간, 가지치기, 2D 격자, 마방 상수, 생성 vs 검증 vs 해 개수 분해.

**시스템(추후)**: Grid / Constraint / Validator 경계; CLI·대화형·API는 미정이나 동일 규칙; n×n 확장 가능성.

**상황 문장**: 개발자는 수많은 배치 중 합이 맞는 배치의 존재·획득·올바름을 **반복 가능·설명 가능한 절차**로 옮기려 하며, 지금은 그 절차의 목적이 열린 상태에서 **상황을 고정**하려 함.

| 이유 | 교육 규모; 짝수 차 대표 |
| 맥락 | 알고리즘/구조화 학습 + 추후 도구; 빈 프로젝트 |

# STEP 2 — Why #1

## 가정한 답 (A)

> **4×4에 1~16을 한 번씩 넣었을 때, 행·열·대각선 합이 같은 ‘올바른 배치’를 하나 확보해야, 진짜 마방진인지 검증·제출·비교할 수 있다.**

**완성 = 제약을 만족하는 해를 갖는 것**; 그래야 맞다/틀리다를 말할 수 있음.

## 불편함

1. **완성 기준 애매** — 어떤 선까지? 해 1개 vs 전부? 완성 vs 검증만?
2. **목적 vs 수단** — 완성은 검증·제출용 수단인데 완성 자체가 목표처럼 들림 (학습/과제/퍼즐별 평가 충돌).
3. **4×4 완성 난이도** — 짝수 차, 수작업 어려움; 수단(수동/탐색/절차) 미정.
4. **해 1개의 약한 증명** — 인스턴스 검증 ≠ “항상 맞는 생성”; 겉보기 완성 vs 정의상 완성.

## 구조적 문제

- **Solution-first, Constraint-second** — 16칸 채우기에 치우침.
- **단일 해 가정** — 동치·해 개수 미반영.
- **Who completes?** — 사람 / 프로그램 / 외부 입력 불명.
- **Outcome vs Process** — 해 1개 출력이 학습 목표와 어긋날 수 있음.
- **비기능 부재** — 시간·확장(n×n)이 Why에 없음.

## Why #1 정리

| 항목 | 내용 |
|------|------|
| 가정 A | 배치 1개 → 검증·제출·비교 |
| 핵심 불편 | 완성의 정의·범위·주체·개수가 열림 |
| 핵심 구조 문제 | 목표·제약·학습·신뢰가 한 문장에 뭉개짐 |

**STEP 3 전 하위 질문**: 완성=생성/검증? 누가? 해 몇 개? 어떤 선?

## 3. 오류 방지

갭: Human error vs Software defect; 4×4만이면 손검증도 가능 → 탐색·다수 후보와 묶을 때 강함.

# STEP 4 — Why #3 (핵심 발견)

TDD = 기능 먼저가 아니라 **무엇이 맞는지(통제·불변·입출력)를 먼저 고정**한 뒤 절차를 채우는 순서.

## Why #3 한 문장

마방진은 **16칸 채우기**가 아니라 **검증 규칙·제약·불변의 층**; STEP 2의 완성 정의 미정·검증 누락·잘못된 확신을 막으려 **행동(생성/탐색)보다 판정 기준을 테스트로 고정**.

## 1. 무엇이 통제되어야 하는가

| 통제 대상 | 실패 증상 |
|-----------|-----------|
| 검증 범위(선) | 정의상 틀림/과도 요구 |
| 도메인(1~16, 4×4) | 중복·누락 통과 |
| 마방 상수 M=34 | 기준 혼동 |
| 생성 vs 검증 | 잘못된 격자를 맞다고 함 |
| 입력·출력 의미 | 침묵 오류·오해 |
| 결정성 | 같은 입력 다른 판정 |
| 탐색 경계 | 무한/과다 출력 |

**순서**: 불변·검증·입출력 계약 → 부분 구현 → 생성/탐색.

## 2. 불변 조건

**정의상**: 4×4; 원소 ∈{1..16} 상호 다름; 명세된 각 행·열·(선택)대각 합 = 34.

**핵심 발견**: **집합 불변 + 선형 불변** 교차 — 한쪽만이면 그럴듯한 오답 통과.

**계층**: L0 형식 → L1 {1..16} → L2 선 합 = M → L3 출력·동치 정책.

**동치**: “마방진이다” vs “출력을 정규형 하나로” — 테스트 전 정책 고정.

## 3. 왜 입출력이 명확해야 하는가

테스트 = **계약**. 모호하면 명세가 아니라 구현 설명.

- 입력: 격자 형태, 빈 칸, 부분 격자, 잘못된 크기 시 실패 관례.
- 출력: 수용/거부/이유/배치 의미 분리; 동치·순서·개수 보장.

**명확화 = TDD 가능 경계 = 생성/검증 뭉개짐 해소.**

# STEP 5 — 진짜 문제 정의

## 1. 표면 문제 정의 (잘못된 정의)

> **4×4 칸에 1부터 16까지 숫자를 넣어서, 행·열·대각선 합이 같은 마방진을 완성하는 프로그램을 만든다.**

**함정**: 행위 중심; 성공 조건·선·값 집합 누락; 산출물·주체 혼동; 도구=목표; 16칸 퍼즐 착시.

## 2. 개선된 문제 정의 (정확한 정의)

> **4×4 격자 배치가 사전 합의한 마방진 규칙(값 집합·선 집합·마방 상수)을 모두 만족하는지 판정하고, 필요 시 같은 규칙으로 확보·비교할 수 있도록, 판정 기준을 명시적·재현 가능·반복 적용 가능하게 다루는 문제이다.**

**요소**: 대상 4×4; 값 {1..16} 각 1회; M=34; 명세된 선 합 = M; 활동 = 판정·반복·확보·비교; 품질 = 동일 입력 동일 판정.

**미결**: 부대각 포함, 해 개수, 배치 주체 — **사전 합의** 필요.

## 3. 핵심 Invariant

**구조**: I-Shape, I-Cell, I-Range, I-Set  
**수치**: I-Constant(34), I-Line-Row/Col/Diag(명세 종속)  
**운용**: I-Judge-Complete, I-Judge-Repeat, I-Separation, I-Equivalence  

**의존**: Shape→Cell→Range→Set→Constant→Line-*→Judge-Complete→Repeat/Separation/Equivalence  

**발견**: 집합 불변 + 선형 불변 **동시** 성립.

## 4. 훈련하려는 사고 능력

1. 규칙 층위화  
2. 판정 우선  
3. 명세적 사고  
4. 불변식 사고  
5. 계약·경계  
6. 재현·반복 태도  
7. 오류·확신 구분  

**주 목표 아님**: 배치 암기, 손속도만, 도구 사용 자체.
